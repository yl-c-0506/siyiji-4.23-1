# 思忆集 app 与底座分离设计

本文档用于收敛一个关键设计原则：思忆集客户端必须可以在没有后端底座时独立工作，而 `automation-hub` 作为统一底座负责为客户端提供增强能力，而不是反向绑死客户端。

主窗口拆分与迁移细节已单独维护在：

- [main-window-decoupling-plan.md](main-window-decoupling-plan.md)

这份文档回答四个问题：

1. app 与底座分别负责什么。
2. 哪些能力必须本地可用，哪些能力适合沉到后端。
3. 客户端、插件、悬浮助手、工作台和同步之间 如何分层。
4. 后续实施时应按什么顺序推进，避免架构再次耦合失控。

## 设计目标

- 保持思忆集 app 在无网络、无登录、无底座情况下仍然可用。
- 保 持 `automation-hub` 作为唯一后端底座，负责同步、身份、备份、审计、插件分发和跨设备能力。
- 避免“桌面端只是一个薄壳、所有能力都必须依赖云端”的错误演进。
- 为可布局工作台、悬浮助手、插件体系和后续 Android 端提供稳定边界。

## 核心原则

1. 本地优先，云端增强。
2. 交互留在客户端，平台能力留在底座。
3. 结构化数据与文件内容分层。
4. 插件通过接口接入，不直接依赖窗口对象和底层存储。
5. 悬浮助手只是入口层，不承接底层业务实现。
6. 同步失败不应阻断本地使用。
7. 设备端可以自治，长期资产语义不能自治。

## 2026-04-20 补充：双 端自治与语义单合同

本次重规划后，app 与底座分离除了继续坚持“本地优先，云 端增强”，还要固定“双端自治、语义单合同”的约束：

- 手机和桌面都允许拥有各 自的本地数据库、本地模型入口和本地工作流。
- automation-hub 继续承接同步、身 份、审计、备份和跨设备能力，但不再被视为唯一认知主脑。
- 长期记忆、任务状态、收件箱、执行回执、偏好和设备记录必须共用同一套对象合同与同步语义。
- 第三方模型会话线程、向量索引、临时摘要和本地缓存都视为派生层或外部引用，不承担长期真源职责。
- 桌面端的职责应正式收口为“重型工作台 + 高级控制台 + 冲突与恢复入口”，而不是语义中心。

## 2026-04 补充：隐私分级与长期恢复责任

为了确保项 目长期围绕“数据主权 + 可迁移架构”演进，app 与底座分离除了功能边界，还要固定数据 边界和恢复责任。

### 隐私分级

平台至少固定四档数据等级：

1. 仅本地：只允许保留在本机，不进入同步、分享和第三方模型。
2. 可同步：允许进入底座同步与跨设备恢复，但默认不分享、不出第三方模型。
3. 可分享：允许在用户显 式操作下导出或分享给其他人或外部系统。
4. 允许送第三方模型：只有显式授权后， 才允许发送给外部 provider 做推理。

要求：

- 默认最小外发；未标记为“允许送第三方模型”的内容，不应被手机直连 provider、桌面 provider 或后端代理默认 带出系统边界。
- 隐私等级属于数据元属性，不能只靠 UI 提示或口头约定。
- 高敏感内容在无显式授权前，只允许本地处理或进入自托管推理路径。

### 长期恢复责任

app 与底座必须共同承担“长期可恢复”责任，而不是把恢复能力外包给某个时代的单一外部服务。

- app 侧至少保证本地快照、导出入口、迁移兼容和离线可读。
- 底座侧至少保证版本化同步、审计追溯、备份编排、恢复演练和跨设备一致性。
- 摘要、索引、向量、缓存都视为可重建派生层；真正需要长期恢复的是结构化资源、附件内容、偏好、任务状态和审计记录。
- 百度网盘、对象存储或未来其他外部平台可以作为备份目的地，但不能反向定义主数据模型、恢复流程和唯一导出格式。

这一补充的核心不是“多做几份备份”，而是确保换模型、换设备、换云厂商甚至换本地存储实现时，用户仍能从同一套数据与状态语义继续使用系统。

## 当前阶段的额外约束

结合当前产品主线，app 与底座分离在 2026-04 还需要额外坚持下面几条约束：

1. 浏览器扩展优先服务于“结构化网页上下文采集 -> 动态收件箱 -> 当前内容整理”， 不单独演进成高权限浏览器产品。
2. 小问角色运行时属于 app 上层入口，不应绕过底层的动作治理、权限矩阵和审计链路。
3. 不把 QQ / 微信等聊天输入扩张成平台主舞 台；它们当前只属于受控输入源候选。
4. 设备配对、同步状态和冲突恢复优先做成可 靠产品能力，再扩大更重的个人云协同能力。

## 运行模式

### 模式 A：本地独立模式

适用场景：

- 用户未配置底座地址
- 用户离线使用
- 用户只需要本地个人助手能力

该模式下必须可用的能力：

- 工作台与模块布局
- 笔记、任务、提醒、翻译历史、单词本、新闻收藏等本地核心数据
- 悬浮助手
- 本地提醒与快速记录
- 插件启停与本地插件能力

该模式下可以降级的能力：

- 多设备同步
- 统一身份
- 审计留痕
- 云备份与恢复
- 插件远端分发

### 模式 B：底座增强模式

适用场景：

- 用户配置了 `automation-hub` 地址和身份凭证
- 用户需要跨设备同步、云备份、统一身份和平台化能力

在该模式下新增的能力：

- 统一身份与设备管理
- 结构化数据同步
- 备份与导出
- 审计与可追溯性
- 插件分发与后续跨端能力

要求：

- 底座能力失败时，客户端仍回退到本地可用状态
- 不允许因为同步或鉴权失败导致主工作流不可用

## 当前建议的推进顺序

从分层角度看，当前 最合理的实施顺序不是“先把所有层都铺满”，而是：

1. 先把入口与交互层稳定下 来，尤其是主窗口、托盘、悬浮入口和桌面 Doctor。
2. 再把客户端应用层中的工作台编排、快速收件、当前内容整理和低打扰建议做成稳定主线。
3. 然后收口数据与能力 适配层，减少 UI 直接碰远端 API、本地 JSON 或历史兼容路径。
4. 接着再强化同步 可靠性、设备概念、冲突处理和云端增强能力。
5. 最后才扩大插件分发、个人内容云 和更重的跨端能力。

这样推进的原因很简单：如果入口层和适配层都还不稳，越早扩大底座能力，越容易把桌面端重新绑死到后端状态上。

## 模块边界

### 一、客户端入口层

包含：

- 主工作台
- 悬浮助手
- 各业务对话 框
- 后续移动端和 Web 入口

职责：

- 展示信息
- 承接用户操作
- 分发导航和快捷入口

不负责：

- 直接操作远端 API 细节
- 直接读写底层存储格式
- 直接实现同步策略

### 二、客户端应用层

包含：

- 工作台编排器
- 悬浮助手控制器
- 插件宿主
- 本地服务对象
- 视图模型或控制器

职责：

- 组织页面和模块
- 维护界面级状态
- 将用户动作转成领域服务调用

不负责：

- 直接持有 SQLite/JSON 细节
- 直接编写网络请求逻辑
- 直接混入第三方平台密钥

### 三、数据与能力适 配层

包含：

- 本地仓储接口
- 远端仓储接口
- 同步调度器
- 插件接口定义
- 能力门面层

职责：

- 向上提供稳定的领域接口
- 向下隔离本地存储和远端 API 差异
- 承担同步重试、冲突处理、状态归一化

 这是整个“app 与底座分离”的关键层。客户端上层不应绕过这一层直接碰本地文件或远端请求。

### 四、本地持久层

包含：

- 本地主存
- 本地缓存
- 工作台布局配置
- 插件配置
- 本地习惯与偏好数据

职责：

- 保障离线可用
- 保存快速访问数据
- 保存用户个人化配置

建议方向：

- 长期以 SQLite 为主存
- JSON 逐步降级为迁移输入、导出格式或轻量缓存

### 五、底座能力层

由 `automation-hub` 承接，负责：

- 身份与设备
- 数据同步
- 云备份与恢复
- 审计与追溯
- 插件分发
- 文件内容云
- 搜索与跨设备能力

底座不负责：

- 强制接管客户端 UI
- 决定客户端首页布局
- 直接承接悬浮助手表现逻辑

补充说明：底座可以提供同步摘要、最近继续事项、设备状态和审计结果，但这些信息如何在工作台、悬浮助手和角色层里呈现，仍由 app 决定。

## 领域能力归属

### 必须优先保留在 app 的能力

- 主界面工作台布局
- 悬浮助手显示与交互
- 本地提醒
- 快速记录
- 本地缓存和离线浏览
- 用户个性化界面偏好

### 适合沉到底座的能力

- 登录与设备身份
- 多端同步
- 云备份
- 文件上传下载
- 审计日志
- 统一搜索
- 插件分发

### 需要双端配合的能力

- 笔记、任务、提醒 、翻译历史、单词本、新闻收藏
- 主界面布局配置同步
- 常用 app 清单和工具偏 好
- 插件配置
- 最近继续事项和跨设备恢复

原则：

- 数据编辑体验在本地完成
- 同步、审计、跨设备一致性在底座完成

## 工作台、悬浮助手 、插件三者关系

### 工作台

工作台是主入口，负责：

- 模块编排
- 模块布局
- 常用工具入口承载
- 同步状态的主展示位置

### 悬浮助手

悬浮助手是轻入口，负责：

- 快速回到工作台
- 快速记录
- 提醒提示
- 同步摘要提示
- 后续桌宠化入口

它不应该直接拥有完整业务实现，而应调用工作台和领域服务暴露出来的统一能力。

### 插件体系

插件体系是扩展机制，负责：

- 让首页卡片可扩展
- 让快捷动作可扩展
- 让数据源和悬浮助手入口可扩展

插件不应直接访问窗口对象和底层数据库，而应通过插件接口调用能力门面。

## 通信关系

### 客户端内部通信

推荐关系 ：

- 入口层 -> 应用层 -> 能力适配层 -> 本地/远端仓储

不推荐关系：

- 对话框直接调用远端 API
- 悬浮助手直接操作本地 JSON
- 插件直接读取主窗口内部字段

### 客户端与底座通信

统一通过 `automation-hub` API 完成，建议按四类接口分层：

1. 身份接口
2. 同步接口
3. 文件与备份接 口
4. 插件与配置接口

客户端只消费稳定 API 契约，不依赖后端内部实现细节。

## 建议的模块拆分

### app 侧建议模块

- `app_shell`：程序入口、模式判断、生命周期管理
- `workspace`：工作台布局与模块编排
- `floating_assistant`：悬浮助手状态与动作控制
- `plugins`：插件宿主、插件注册、插件元数据
- `services`：本地提醒、快速记录、信息筛选、工具收束等应用服务
- `repositories`：本地仓储和远端仓储抽象
- `sync`：同步调度、冲突处理、状态上报
- `settings`：本地配置、底座连接、工作台偏好

### 底座侧建议模块

- `auth`：用户、设备、token
- `personal-data`：结构化个人数据资源
- `sync-center`：增量同步、冲突策略、设备游标
- `backup`：备份与恢复
- `plugin-registry`：插件元数据与分发
- `content-store`：文件与附件存储
- `search`：全文 检索与统一查询

## 当前最需要压住的反模式

后续设计与实现时，应持续 避免下面几类反模式重新出现：

1. 对话框直接调远端 API。
2. 悬浮助手直接操作本地 JSON 或主窗口内部字段。
3. 新功能继续追加到 `DatabaseHandler`。
4. 角色层绕过权限矩阵和动作治理直接执行高风险动作。
5. 因同步或鉴权失败直接阻 断本地核心编辑与整理流程。
- `audit`：关键动作审计

## 演进顺序

### 阶段 1

- 先把客户端内部边界收紧：入口层、应用层、能力适配层分清
- 停止继续让 UI 直接持有业务状态和网络逻辑

### 阶段 2

- 把工作台布局、悬浮助手和插件接口稳定下来
- 让 app 在本地独立模式下形成可用闭环

### 阶段 3

- 再逐步接入底座增强能力：身份、同步、备份、插件分发
- 同时保 证本地独立模式不被破坏

### 阶段 4

- 扩展到 Android、Web 或更多入口
- 复用相同领域模型、同步协议和底座能力

## 当前最重要的架构约束

1. 不再继续扩大主窗口和对话框的职责。
2. 不再继续扩大 `DatabaseHandler` 这类大而全对象的边界。
3. 不允许插件直接访问主窗口内部状态。
4. 不允许悬浮助手直接承接完整业务逻辑。
5. 不允许底座接入破坏离线可用性。

## 参考项目借鉴与安全补充（2026-04-06）

以下内容来自对 openclaw、nanobot、syncthing、My-Neuro 和七奈桌宠的对标分析，聚焦于"app 与底座分离"这一核心话题。

### 1. 浏览器桥接通道安全

浏览器扩展通过 HTTP 与桌面端通信时（类似 syncthing 的本地 API 设计），必须做到：

| 控制项 | 做法 |
|--------|------|
| 绑定地址 | 强制 `127.0.0.1`，不绑 `0.0.0.0` |
| Origin 校验 | 服务端只接受来自浏览器扩展 Origin 的请求 |
| 会话令牌 | 首次握手时生成 一次性 session token，后续请求均携带 |
| 输入净化 | 所有来自扩展的 HTML/文本经过白名单标签过滤后 再入库 |

> syncthing 在本地 API 上同时使用 API Key + CORS 白名单 + TLS（ 可选），值得参考。

### 2. LLM 调用安全边界

借鉴 nanobot 的 Provider Protocol 和 openclaw 的 Tool Use 注册表：

- **系统提示 / 用户输入分离**：不把未经过滤的用户文本直接拼入 system prompt，防止提示词注入。
- **LLM 返回值结构化校验**：LLM 输出在执行任何动作前，必须经过 schema 校验（如 JSON Schema  或 Pydantic model_validate），不通过则拒绝执行。
- **action_guard 网关**：openclaw 对 shell/browser 等高风险工具使用权限网关；思忆集的 `action_guard.py` 承担相同职责，LLM 建议的所有副作用性操作（启动应用、文件操作、网络请求）均须经过该网关确认。

### 3. 数据库演进策略

当前阶段 SQLite 是正确选择，但需要 提前准备迁移路径：

| 阶段 | 做法 |
|------|------|
| 当前 | 启用 WAL 模式，减少读写互斥 |
| 短期 | 引入 alembic 管理 schema 版本，停止手动 DDL |
| 中期 | 仓储接口后面同时放 SQLite 和 PostgreSQL 适配器 |
| 长期 | 底座侧切 PostgreSQL，app 侧保持 SQLite 本地缓存 |

### 4. 同步协议设计参考

syncthing 的 Block Exchange Protocol 提供了关键启示：

- **增量同步**： 以内容哈希（而非时间戳）判断变更，避免时钟漂移问题。
- **冲突解决**：写入冲突时保留双方副本（`.sync-conflict-*`），让用户手动合并而非静默覆盖。
- **设备发现**：syncthing 通过 Local Discovery + Global Discovery 实现设备配对。思忆集可以先实现"扫码配对 + 手动输入底座地址"的简化版本。

### 5. 与"反模式"列表的联动

上面"当前最需要压住的反模式"列表中，应补充一条：

> **6.  浏览器桥接通道不做 Origin 校验和会话令牌校验，导致本地任意页面可调用桌面端能力。**

这一条在扩展功能推进之前必须堵住，否则会在本地宿主层暴露远控面。
