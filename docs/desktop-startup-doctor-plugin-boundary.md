# 思忆集桌面端首次引导、Doctor 与插件边界

本文档用于把桌面端最近新增的三条产品化入口收敛成正式约定：

1. 首次启动引导
2. 统一 Doctor 诊断入口
3. 内部插件扩展边界

本文件只保留边界与产品化要求；Doctor 调试资产的结构化模型与执行计划已拆到：

- [desktop-doctor-debug-assets.md](desktop-doctor-debug-assets.md)

## 1. 首次启动引导

目标不是做一次性欢迎弹窗，而是把“首次可用”收敛成稳定入口。

当前行为：

- 主窗口首次启动且未完成引导时，会自动弹出“首次启动引导”
- 如果处于 `SIYI_NONINTERACTIVE_STARTUP=1` 或 `QT_QPA_PLATFORM=offscreen`，引导不会弹出，避免干扰 smoke 和冷启动检查
- 完成后会写入 `startup_guide_completed=true`
- 后续仍可从“设置 -> 首次启动引导”再次打开

引导关注的最小闭环：

- 主 AI Provider 已配置
- 备用或本地回退 Provider 至少有一条可选退路
- automation-hub 地址与 Token 已配置或明确当前不启用
- 桌面 Doctor 已跑过一次

## 2. 桌面 Doctor

统一入口：

- UI：菜单“设置 -> 桌面 Doctor”
- CLI：`python scripts/desktop_doctor.py [--deep] [--json] [--debug-assets-json]`

CLI 参数约定：

- `--json`：输出原始诊断 report JSON，保持兼容语义
- `--debug-assets-json`：输出结构化调试资产 JSON，用于 UI 复用、排障与复现包摘要

当前固定检查项：

- 入口脚本是否存在
- 数据目录是否存在、核心 JSON 文件是否缺失
- LLM Provider 三层链路拓扑
- automation-hub 地址是否配置，深度模式下是否连通
- 浏览器桥是否启用、端口是多少、当前运行态如何
- 自动同步开关状态
- 七奈桌宠目录、启动器和便携 Java 是否存在
- 命令白名单是否缺失关键项
- 当前版本通道
- 插件扩展点与已注册数量

深度模式增加：

- LLM 连通性测试
- automation-hub `/health` 与可选 `/auth/me` 验证

当前建议把 Doctor 继续朝“桌面端统一诊断与修复入口”推进，而不是只做静态信息展示。后续优先级建议固定为：

1. 先给出明确结论：能否启动、能否同步、能否桥接、能否恢复。
2. 再给出失败原因归类：配置缺失、服务未启动、凭据不可用、端口冲突、状态损坏、兼容路径残留。
3. 最后再补修复建议和可自动修复项，而不是先堆更多检查项。

补充约定：Doctor 的后续 UI 和工作台接入应统一消费结构化调试资产，不再各自重建结论、归因和建议逻辑；详见 [desktop-doctor-debug-assets.md](desktop-doctor-debug-assets.md)。

当前状态补充：Doctor CLI 已支持 `--debug-assets-json`，Doctor 对话框也已支持直接导出结构化调试资产 JSON，二者共用同一份模型。

补充说明：这部分可以借鉴 `openclaw doctor` 的思路，但不照搬其消息平台与服务守护模型。思忆集当前更需要的是桌面端入口、配置、桥接、同步和状态恢复的统一诊断能力。

## 3. 插件边界

当前不做开放市场，先把内部解耦边界固定下来。

正式支持的四类扩展点：

1. 首页卡片
2. 快捷动作
3. 悬浮助手扩展
4. 数据源接入

对应协议入口：

- [思忆集app/plugin_protocol.py](../思忆集app/plugin_protocol.py)

当前原则：

- 新功能优先接入这四类扩展点，不再直接堆进主窗口
- 插件注册中心统一维护元数据和数量统计
- Doctor 负责输出当前扩展点与注册情况，避免插件边界变成隐式约定

当前阶段还应补充三条硬约束：

1. 不把插件系统做成任意脚本执行平台。
2. 不允许插件无限制直接访问窗口对象、本地数据库和敏感配置。
3. 插件优先服务内部解耦，不以开放第三方市场为当前目标。

建议的产品化顺序：

1. 先把首页卡片和快捷动作两类插件用起来。
2. 再补悬浮助手扩展点和数据源接入点。
3. 最后才考虑插件配置页、安装流和内部插件分发。

## 4. LLM Provider 约定

当前桌面端统一采用三层链路：

1. 主 Provider
2. 备用 Provider
3. 本地回退 Provider

对应拓扑入口：

- [思忆集app/llm_provider_topology.py](../思忆集app/llm_provider_topology.py)

要求：

- 设置页必须可查看和修改三层配置
- 诊断入口必须能输出拓扑摘要
- 后续新的 AI 调用链路优先复用这套拓扑，不再各自散落判断主 / 备用 / 本地配置

## 5. 版本通道

当前统一使用 `assistant_upgrade_channel` 表达版本通道。

支持值：

- `stable`
- `preview`
- `dev`

约定：

- `stable` 用于日常可用版本
- `preview` 用于预发布验证
- `dev` 用于持续开发和联调

## 6. 当前阶段最小完成定义

如果把首次引导、Doctor 和插件边界视为一组产品化入口，当前最小完成定义建议固定为：

1. 首次启动后，用户能在一次引导中完成 Provider、底座连接和 Doctor 初检。
2. Doctor 至少能输出一份用户可理解的结论，而不是只有技术状态明细。
3. 两类以上内部模块已通过插件接口接入，而不是继续只靠主窗口硬编码。
4. 配置迁移、引导结果和 Doctor 结论在设置页里能被重新查看与触发。
