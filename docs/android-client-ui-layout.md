# 思忆集 Android 客户端 UI 布局设计

本文档用于把“手机端 UI 要像小爱唤出态”这件事落到可实现的界面规范上。

参考目标不是逐像素复刻系统级小爱，而是借用它的三个核心体验：

- 以底部唤起层为主，而不是传统满屏聊天页。
- 在桌面、文件、截图、语音之间切换时都保持同一套助手容器。
- 让输入、快捷动作、附件入口、语音入口始终处在手指最容易够到的位置。

本文档是 Android UI 母版，不是当前信息架构真源。涉及“ 当前哪些页面已经存在、当前叫什么、当前能直接开发哪些页面”，以 [docs/android-client-technical-design.md](android-client-technical-design.md) 第 0 节和第 13 节为 准。

## 0. 使用方式与状态标签

为避免把未来页面直接当成当前需求，本文档固定使用三个状态标签：

- `Current`：当前代码已经有对应页面或明确的当 前路由，可直接按本文补 UI 细化。
- `Transitional`：当前代码已有入口，但语义仍是过渡态，只能做局部体验优化，不能冒充正式产品页。
- `Future`：交互母版或未来页面方向，当前不能默认直接开做。

### 0.1 当前代码页面与概念页映射

| 当前代码页 / 路由名 | 对应概念页 | 状态 | 说明 |
| --- | --- | --- | --- |
| `Home` / `AssistantHomeScreen` | 默认唤起页 | `Current` | 当前首页已经承担 资源入口与快捷收件 |
| `Chat` / `ChatScreen` | 轻会话页 | `Transitional` | 当前仍是 `personal note -> agent/ask` 过渡链路 |
| `FileTransfer` / `FileTransferAssistantScreen` | Quick Capture 页 | `Transitional` | 当前不是正式文件传输， 只是文本/剪贴板轻写入 |
| `NotesList` | 笔记列表页 | `Current` | 只读 |
| `TasksList` | 任务列表页 | `Current` | 只读 |

| `Wordbook` | 词库页 | `Current` | 只读 |
| `SyncCenter` | 同步中心页 | `Current` | 当前是资源级状态摘要，不是完整增量同步控制 台 |

### 0.2 当前禁止误读的点

- 本文提到的“文件传输助 手页”，如果没有特别声明，默认指的是未来的助手化文件云交互母版；当前代码中的 `FileTransfer` 先解释为 Quick Capture。
- 本文提到的晨报卡、识屏助手页、通知详情 、真正文件传输会话页，默认都是 `Future`，除非在 [docs/android-client-technical-design.md](android-client-technical-design.md) 的当前矩阵里升级。
- 本文定义视觉语言和交互方向，不单独定义后端接口、对象字段和当前页面集合。

## 1. UI  定位

思忆集 Android 客户端首页不应优先做成“多标签应用壳”，而应优先做成“助手唤起层 + 卡片工作流”。

这意味着：

- 默认主入口是底部助手条。
- 常见任务通过快捷动作触发，而不是先进入复杂菜单。
- 需要深入编辑时，再进入笔记、任务、文件同步等全页页面。

## 2. 参考图对应的交互状态

用户给出的三张图，实际对应三个状态。

### 2.1 文件对话态

对应第一张图。

特征：

- 顶部是当前上下文标题，例如“文件传输助手”。
- 中间是消息流、文件卡片、图片缩略图。
- 底部是高可达性的输入栏。
- 输入栏右侧固定保留语音和加号入口。

这个状态适合：

- 文件同步助手
- 个人助手问答
- 笔记转任务
- 文章总结

### 2.2 附件与快捷动作菜单态

对应第二张图。

特征：

- 点击加号后，不跳转新页面，而是在底部输入栏上方弹出短菜单。
- 菜单项目是高频入口，不宜过多。
- 项目应该支持图标 + 文案，点击后 直接进入对应动作。

这个状态适合：

- 相机
- 相册
- 识屏
- 文档
- 屏幕翻译
- 文件同步
- 扫码配对

### 2.3 桌面唤起态

对应第三张图。

特征：

- 背景仍然看得见桌面，不是完全切走。
- 底部是悬浮输入条，具有系统助手气质。
- 输入条上方可以叠加少量建议卡片或快捷提示。
- 整个界面是“半浮层”，而不是沉重的应用首页。

这个状态适合：

- 快速记一条想法
- 说一句“同步一下文件”
- 识别当前屏幕并生成笔记
- 快速打开最近任务、最近笔记

## 3. 思忆集手机端的推荐布局

## 3.1 总体结 构

```text
全屏容器
  ├─ 背景层
  │   ├─ 当前桌面/页面截图或应用背景
  │   └─ 半透明暗化蒙层
  ├─ 顶部轻状态区
  │   ├─ 当前场景标题
  │   └─ 场景副标题/同步状态
  ├─ 中部内容区
  │   ├─ 建议卡片
  │   ├─ 最近事项卡片
  │   ├─ 会话流
  │   └─ 文件/图片/任务卡片
  └─ 底部助 手输入区
      ├─ 快捷动作 chips
      ├─ 输入胶囊
      ├─ 语音按钮
      └─ 加号按钮
```

## 3.2 底部输入区是视觉中心

整个手机端布局的核心，不是顶部导航，而是底部助手输入区。

输入区建议由三层组成：

### 第一层：快捷动作 chips

放在输入框上方，默认显示 2 到 4 个高频动作，例如：

- 深度思考
- 文件同步
- 识别当前屏幕
- 新建笔记

这些动作应该是低高度、圆角胶囊按钮，不宜做成大卡片。

### 第二层：输入胶囊

输入栏保持一整条圆角胶囊，视觉上尽量贴近你截图里的小爱唤起条。

建议内容：

- 左侧：场景图标或相机按钮
- 中间：占位文案，例如“发送消息或按 住说话”
- 右侧：麦克风按钮
- 最右：加号按钮

### 第三层：扩展动作弹 层

点击加号后，弹出竖向或 2 列小面板，不切走当前界面。

建议动作：

- 相机
- 相册
- 文档
- 识屏
- 屏幕翻译
- 文件同步
- 扫 码配对
- 从剪贴板导入

## 3.3 中部内容区不做复杂列表首页

为了保 持“小爱唤起态”的轻盈感，中部不要一上来堆满多标签和复杂信息流。

中部应该根据场景切换内容。

### 唤起首页

- 今日建议
- 最近笔记
- 最近待办
- 最近同步状态

### 会话模式

- 用户消息气泡
- 助手回复气泡
- 文件卡片
- 图片卡片
- 执行结果卡片

### 文件同步模式

- 最近传输文件
- 手机到电脑
- 电脑到手机
- 当前同步任务状态
- 冲突提醒卡片

## 4. 页面状态建议

## 4.1 默认唤起页 `Current`

用于从 桌面快速呼出个人助手。

布局建议：

- 顶部只保留很轻的标题，如“思忆 集助手”。
- 中部显示 1 张主建议卡 + 2 行最近事项。
- 底部显示输入条。

主建议卡可轮播：

- 继续昨天未完成的任务
- 检查文件同步状态
- 打 开最近编辑笔记

## 4.2 Quick Capture / 文件传输助手页 `Transitional`

当前代码里，这个入口优先解释为 Quick Capture，而不是正式文件传输协议页。也就是说：

- 当前可做：文本输入、剪贴板导入、设备未接入时跳配对、成功后给出“已 写入个人笔记”的回执。
- 当前不做：真实上传/下载进度、文件大小、来源设备、断点续传、文件打开方式。
- 未来若进入正式文件云阶段，再把这里升级成真正“像聊天一 样收发文件”的助手页。

布局建议：

- 顶部标题：文件传输助手
- 顶 部副标题：显示设备名、在线状态、最近同步时间
- 中部：按聊天流展示文件卡片和图片卡片
- 底部：输入栏 + 加号菜单

文件卡片建议包含：

- 文件名
- 文件大小
- 来源设备
- 上传/下载进度
- 打开方式

如果当前只做 Quick Capture，更适合先用下面这组轻量卡片替代文件卡片：

- 当前输入摘要
- 写入目标提示，例如“将保存到个人笔记”
- 成功/失败回执
- 去配对后回来继续 的提示

## 4.3 识屏助手页 `Future`

从底部加号菜单进入。

布局建议：

- 顶部保留“识别当前屏幕”标题
- 中部展示识别结果卡片
- 结果卡片下面直接给动作按钮：
- 保存为笔记
- 总结内容
- 翻译
- 加入稍后读
- 发到电脑端

## 4.4 同步中心页 `Current`

这是全页，不再保持纯唤 起态，但视觉语言仍延续底部助手体系。

布局建议：

- 顶部展示设备名和账户状态
- 中部按资源分区：
- 笔记
- 任务
- 新闻收藏
- 翻译记录
- 词库
- 文件
- 每个资源展示：
- 上次同步时间
- 待同步数量
- 冲突数量
- 立即同步按钮

当前实现收口要求：

- `Current`：先展示笔 记、任务、翻译记录、词库四类资源的最近同步时间、错误摘要和手动刷新。
- `Future`：新闻收藏、文件、cursor 增量状态、冲突列表、tombstone 诊断面板。

## 4.5 每日简报与轻执行落点 `Future`

如果 Android 第一批要接“每日新闻定时推送”，UI 不应新增一套独立新闻主页，而应把它接入现有助手唤起层。

推荐落点如下 ：

- 默认唤起页顶部增加“今日晨报”主卡。
- 卡片正文展示 2 到 3 条摘要和一个状态标签，例如“08:30 已送达”。
- 卡片底部只保留轻执行按钮，不进入复杂菜单。

建议按钮：

- 播放晨报
- 稍后提醒
- 标记已看
- 发回桌面继续处理

通知点击后的行为建议固定成两级：

- 一级：从系统通知直接拉起助手唤起层，并把焦点落到晨报卡。
- 二级：从晨报卡进入详情页，查看完整摘要、来源和关联任务。

详情页仍应延续同一套底部输入区，不单独做重型资讯流页面。推荐内容：

- 今日主题摘要
- 相关新闻来源列表
- 与历史笔记或任务的关联建议
- 轻执行按钮区

这里的“轻执行”优先做移动本地动作：

- 添加本机闹钟
- 加入稍后读
- 快速生成一条笔记
- 同步回桌面收件箱

当前不建议第一版直接做系统级常驻悬浮窗。更稳妥的顺序是：

- 先把通知卡片 + 助 手唤起层 + 晨报详情页打通。
- 再评估 Accessibility、功耗和误触成本后，决定是 否补系统级悬浮建议层。

## 5. 视觉风格建议

## 5.1 视觉关键词

- 半透明
- 轻模糊
- 大圆角
- 低密度信息
- 底部聚焦

## 5.2 背景

不要使用纯白整页背景作为默认主态。

建议：

- 桌面唤起时保留背景截图并叠加 10% 到 18% 深色蒙层
- 应用内页面使用浅雾灰背景
- 卡片用半透明白或暖白

## 5.3 颜色

建议避免做成系统蓝紫默认 AI 风格。

思忆集更适合：

- 主色：雾白 + 墨灰
- 强调色：柔和青蓝
- 成功色：低饱和绿
- 警告色：琥珀橙

## 5.4 字体层级

- 大标题：用于时间、场 景名、唤起态标语
- 中标题：用于卡片标题
- 正文：用于会话内容和说明
- 辅助：用于同步状态、时间、设备说明

## 7. 关键交互建议

## 7.1 输入栏常驻，不随页面风格切换

无论在：

- 默认唤起页
- 文件助手页
- 识屏页
- 轻会话页

底部输入胶囊都应保持同一视觉语言。

## 7.2 加号菜单优先承接“现实世界输入”

优 先级建议：

- 相机
- 相册
- 文档
- 识屏
- 文件同步
- 扫码配对

不要先塞太多抽象功能。

## 7.3 语音按钮默认就是主要入口

 因为你参考的是小爱唤起态，语音入口不能只是附属图标。

应支持：

- 点击开始录音
- 长按说话
- 录音时波形反馈
- 自动转文字后进入会话流

## 7.4 文件同步不做“网盘首页”，而做“会话化操作”

文件同步第一版不建议长得 像传统文件管理器。

建议做成：

- 用户一句“把这个发到电脑”
- 或从加号选择文档/相册
- 然后助手生成文件卡片和传输进度

这种方式更符合你当 前个人助手的产品方向。

## 8. 与当前项目的对齐建议

这套 UI 布局应该服务于当前已经规划好的 Android 接入路线，而不是独立存在。

建议对应关系如 下：

- 默认唤起页：当前 Android 客户端首页
- Quick Capture / 文件传输 助手页：当前先承接文本、剪贴板轻写入；未来再升级成正式文件同步入口
- 会话页：当前承接轻问答；正式统一任务入口要等 `assistant_task` 落地后再升级
- 同步中心页：当前承接资源级状态摘要；未来再承接完整结构化同步模型

后续实现时应继续对齐：

## 9. 当前最小 UI 集合

如果只按当前代码与路线门槛推进 ，建议先把下面这些页面做稳，而不是继续加未来页面：

1. 助手唤起首页
2. 设备接入页
3. 轻会话页
4. Quick Capture 页
5. 同步中心页
6. 笔记/任 务/翻译/词库只读列表页

如果要做母版层面的下一批候选页，再按顺序补：

1. 识屏结果页
2. 晨报卡与详情页
3. 正式文件传输会话页

这样能先把 当前闭环做稳，再让 UI 母版逐步落地。

## 10. 不建议的做法

- 不建议 第一版直接做成传统 IM 全屏聊天首页。
- 不建议第一版直接做成带底部 Tab 的重应 用壳。
- 不建议第一版把文件同步做成复杂目录树浏览器。
- 不建议第一版把所有功能都塞进加号菜单。

## 11. 结论

手机端思忆集 UI 应以“小爱唤起态” 作为交互母版，但内容层必须是思忆集自己的：

- 输入与语音在底部常驻
- 内容以建议卡、会话流、文件卡片承载
- 文件同步、识屏、笔记、任务都从同一个助手容器进入
- 深层能力再进入独立全页

这样既保留系统助手那种轻、快、顺手的感觉，也不会把思忆集做成另一个臃肿聊天壳。

## 12. 落地到 Compose 的页面与路由建议

为了避免 UI 文档和技术实现脱节，第一版建议把页面路由固定成少量、稳定、可扩展的结构。

### 12.1 Current：当前代码页映射

当前代码还没有采用字符串路由表，而是通过顶层 screen enum 切换。现阶段应先保持这组页面集合稳定 ：

```text
Home
Chat
FileTransfer
NotesList
TasksList
TranslationHistory
Wordbook
SyncCenter
DevicePairing
```

### 12.2 Future：推荐概念路由

```text
assistant/invoke
assistant/conversation/{sessionId}
assistant/file-transfer/{deviceId}
assistant/screen-result/{captureId}
sync/center
briefing/detail/{briefingId}
```

说明：

- `assistant/invoke`：默认唤起页，也是首页。
- `assistant/conversation/{sessionId}`：承接普通对话、笔记转任务、总结等轻会话。
- `assistant/file-transfer/{deviceId}`：承接“发到电脑”“从电脑接收”这类会话化文件同步。
- `assistant/screen-result/{captureId}`：承接识屏结果和后续动作。
- `sync/center`：承接同步状态、失败重试、资源分区管理。
- `briefing/detail/{briefingId}`：承接晨报、提醒、通知详情。

### 12.3 顶层状态容器建议

建议把 UI 顶层状态 统一收口为一个 `AssistantSurfaceState`，避免每个页面各自定义完全不同的壳。

推荐状态维度：

- `scene`：当前场景，如 `invoke`、`conversation`、`fileTransfer`、`screenResult`
- `title`：顶部标题
- `subtitle`：副标题或同步状 态
- `quickActions`：当前场景下可见的快捷动作
- `inputState`：输入栏内容、录音状态、附件状态
- `overlayState`：加号菜单、语音面板、底部弹层是否展开
- `contentState`：中部区实际内容

这样做的目的，是让“同一个助手容器承载不 同业务”真正落实到代码结构，而不是仅停留在视觉描述上。

### 12.4 中部内容区建议的数据模型

中部内容区虽然按场景切换，但建议复用同一组卡片协议。

可统一抽象为：

- `SuggestionCardModel`
- `RecentNoteCardModel`
- `TaskPreviewCardModel`
- `FileTransferCardModel`
- `RecognitionResultCardModel`
- `SyncStatusCardModel`
- `BriefingCardModel`

这样后续无论来 源是本地工作流、桌面端回写，还是 automation-hub 下发，UI 层都只消费统一卡片模型 。

## 13. 首版交互链路建议

第一版不需要把所有能力都同时做深，关键 是先把几条最有产品感觉的链路打通。

### 13.0 Current：当前最重要的真实闭环

在当前代码面，最重要的不是晨报、识屏或正式文件云，而是下面三条已经存在的闭环：

1. 首页快捷收件 -> 若未接入则跳设备页 -> 接入成功后自动回原页补发 。
2. 聊天页发送 -> 若未接入则跳设备页 -> 接入成功后自动回聊天页重发。
3. Sync Center 手动刷新 -> 汇总四类资源状态 -> 给出最近同步时间与失败摘要。

只有这三条闭环稳定后，才值得继续往 UI 母版的未来页扩展。

### 13.1 从桌面 通知进入晨报卡 `Future`

链路建议：

1. Android 收到晨报通知。
2. 用户点击通知。
3. 应用直接拉起 `assistant/invoke`。
4. 焦点落在“今日晨报”主卡。
5. 用户可执行：播放、稍后提醒、标记已看、进入详情。

这样能保证 通知体验和助手唤起层是同一套入口，而不是两套割裂页面。

### 13.2 从加号进 入识屏结果 `Future`

链路建议：

1. 用户点击输入栏加号。
2. 展开 扩展动作弹层。
3. 点击“识屏”。
4. 进入识屏处理中间态。
5. 成功后跳到 `assistant/screen-result/{captureId}`。
6. 用户直接执行“保存为笔记”“总结内容”“ 发到电脑”。

这里要特别避免跳进复杂设置页，否则会破坏唤起态的连续感。

### 13.3 从一句话触发正式文件同步 `Future`

链路建议：

1. 用户在唤起页说“把这个发到电脑”。
2. 助手识别当前上下文是图片、文档或剪贴板内容。
3. 自动进入文件传输会话页。
4. 中部出现待发送文件卡和进度状态。
5. 成功后回显设备、时间和打开入口。

这条链路优先体现“助手化同步”，而不是传统文件管理器思路。

### 13.4 从晨报详情转成可继续处理事项 `Future`

链路建议：

1. 用户从晨报卡进入详情页。
2. 在详情页选择“生成笔记”或“加入稍后读”。
3. 动作结果不只显示 toast，而应生成回执卡片。
4. 回执卡片可继续点击进入对应笔记或收件箱事项。

这样能把消费信息和沉淀资产串起来，符合思忆集的产品方向。

## 14. 第一版实现优先级

建议不要按“页面数量”排优先级，而要 按“闭环强度”排。

### P0：必须先通

## 15.  第一版验收标准

如果第一版要判断是否达到“像助手而不是像功能菜单”，可以用下面几条做验收。

### 15.1 交互验收

- 用户从通知进入应用后，3 秒内能 看到可操作主卡和输入栏。
- 用户从首页到底部输入、加号菜单、语音入口的操作，不需要跨页面跳转超过 1 次。
- 用户完成“识屏并保存为笔记”不超过 3 步。
- 用户完成“发送文件到电脑”时，全程都停留在同一助手容器视觉体系内。

### 15.2 视 觉验收

- 输入栏始终是视觉中心。
- 中部内容密度明显低于传统聊天首页。
- 卡片风格、圆角、留白和按钮风格在四个核心页面中保持一致。
- 顶部信息轻，不喧宾夺主。

### 15.3 产品验收

- 用户能够理解首页不是“功能目录”，而 是“可直接行动的助手入口”。
- 晨报、识屏、文件同步、笔记继续处理四条链路能互相衔接。
- 至少有一条通知 -> 助手唤起 -> 轻执行 -> 资产沉淀的完整闭环。
