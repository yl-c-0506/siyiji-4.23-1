# 思忆集下一阶段产品边界

本文档只回答一个问题：

在当前阶段，思忆集到底应该先做什么，不应该急着做什么，以及哪些事情即使用户有兴趣也不应现在进入主线。

这份文档用于约束产品范围，避免项目再次回到“想做的都想做，但没有一条真正闭合的主线”的状态。

## 2026-04-20 当前收口范围

本节是当前阶段的硬边界，优先级高于本文后续仍保留的更广产品候选项。

### 当前正式定位

思忆集当前不再按“多功能桌面原型集合”来定义，而是按两层结构收口：

- 上层只有小问代理层，负责理解目标、跨能力编排、主动提醒和长期偏好。
- 下层只有能力市场，负责目录、权限、健康、启停和调用。

新闻、笔记、翻译、浏览器桥接、同步器、规则中心、子代理和 MCP 工具，当前都应被视为小问下层能力，而不是彼此并列的主产品入口。

### 当前硬范围

当前阶段只做第一期产品收口，不加新大功能：

1. 统一入口固定成聊天问答、执行任务、创建自动化三类。
2. 把 [../思忆集app/assistant_user_profile.py](../思忆集app/assistant_user_profile.py) 设为偏好真源，把 [../思忆集app/assistant_memory_index.py](../思忆集app/assistant_memory_index.py) 退回证据层。
3. 把能力市场先做成目录页和治理摘要，不做安装商店。
4. 把定时与主动结果统一落到 `inbox` 和 `notification`。
5. 守住现有测试面和主链稳定，不为了扩功能破坏统一入口。
6. 手机端在当前阶段只冻结统一对象合同与同步语义，不启动全量 UI 扩面；当前允许切片以 [android-client-technical-design.md](android-client-technical-design.md) 的当前矩阵和 [siyi-m0-m3-implementation-checklist.md](siyi-m0-m3-implementation-checklist.md) 的手机端工作包为准。

### 当前明确延期

- 第二期的偏好进化、反馈写回、任务模板和自动化中心。
- 第三期的 Capability Descriptor、健康状态、权限声明和真正的能力市场。
- 更多外部推送通道、开放插件市场和更重的主动打断能力。

### 当前完成定义

1. 一句话输入可以稳定分流到 chat / execute / automate。
2. 规则输出和主动结果只走 `inbox` 与 `notification` 两类主通道。
3. 设置页能看到画像偏好和能力目录摘要。
4. 偏好真源与证据层的职责已经明确分开。
5. 手机端合同冻结边界已经明确，未进入当前矩阵的未来页面和路由不提前扩面。
5. 现有测试面继续成立，而不是通过关闭测试换取收口表面完成。

## 当前阶段定义

思忆集当前阶段不再是“多功能桌面原型集合”，而是：

一个本地优先、可云端增强的个人动态收件箱、整理器和工作台客户端。

当前阶段的核心目标不是继续扩展功能数量，而是把下面四条主线做成稳定闭环：

1. 个人动态进入思忆集。
2. 动态被整理进私人空间。
3. 思忆集结合上下文给出低打扰建议。
4. 思忆集在明确授权下帮助执行受控动作。

## 当前阶段必须坚持的原则

1. 本地优先，云端增强。
2. 先收口主线，再扩展支线。
3. 先做低风险建议，再做高风险执行。
4. 先做受控自动化，再谈自治自动化。
5. 所有自动动作都要可关闭、可追踪、可回退。

## 当前阶段的执行顺序提醒

本阶段虽然覆盖多个方向，但实际推进顺序应固定为：

1. 先把输入稳定收进来，再谈更复杂的建议和执行。
2. 先把私人空间整理闭环做出来，再扩大来源和格式。
3. 先让工作台和悬浮助手承接“继续入口”，再强化角色化表现层。
4. 先让同步变得可信、可解释、可恢复，再讨论更重的个人云能力。
5. 先走“提案、备份、执行、验证、报告”的监督式自优化，再谈更强自治能力。

补充约束：当前若与本文下文更广的“必须做”候选项冲突，以本节“2026-04-20 当前收口范围”为准；更广的输入、整理、工作台和建议能力，只有在第一期产品收口完成后才恢复为主线推进对象。

补充约束：

- 浏览器扩展当前应服务于“结构化网页上下文采集 -> 收件 -> 整理”，而不是升级成高权限网页抓取器。
- QQ / 微信等聊天输入当前只作为受控输入源候选，不作为产品主舞台。
- 角色化桌宠、皮肤和复杂动画只在悬浮助手最小闭环稳定后再进入主线。

## 一、必须做

这些能力属于下一阶段主线，应该优先进入正式计划。

### 1. 个人动态收件箱

目标：把用户的日常输入统一收到思忆集，而不是继续散落在多个窗口和临时记录里。

本阶段建议优先支持的输入源：

- 当前网页
- 当前文档（PPT、PDF、Markdown、Word 等）
- 剪贴板
- 手动粘贴内容
- 系统通知级消息摘要
- 用户主动选择的 QQ / 微信内容

当前阶段不要求“一次性全量抓取所有聊天历史”，而是先把“可访问、可理解、可落地”的输入收进统一 inbox。

### 2. 私人空间整理器

目标：把原始输入转成可回看、可沉淀、可继续处理的个人记录。

当前已给定的私人空间路径：

- `你的同步目录/思忆集/忆集/日记`

建议在该路径下分三层结构：

- `inbox`：原始采集结果
- `daily`：按天整理后的日记
- `insights`：结构化洞察、主题、待跟进事项

本阶段至少要做到：

- 新输入可以落到 inbox
- 可生成当天整理摘要
- 可把内容转为笔记、任务或后续待处理项

### 3. 当前内容整理能力

目标：针对当前正在看的网页、PPT、PDF 或文本，提供统一的“整理当前内容”动作。

输出应统一支持：

- 摘要
- 要点
- 关键词
- 待行动项
- 思维导图提纲
- 与历史记录的关联建议

重点不是支持多少文件格式，而是把“整理当前内容”做成一个稳定动作入口。

### 4. 上下文建议助手

目标：在有限白名单范围内理解用户当前上下文，并给出低打扰建议。

本阶段允许进入主线的建议类型：

- 总结建议
- 任务化建议
- 笔记归档建议
- 与历史内容的关联建议
- 学习资料的提纲和复盘建议

建议方式应以“提示是否执行”为主，而不是默认直接替用户执行。

### 5. 工作台化首页

目标：把主界面从固定首页推进为工作台容器，而不是继续增长为功能集装箱。

下一阶段必须具备：

- 模块显示与隐藏
- 模块顺序调整
- 至少两种布局模板
- 工作台配置持久化
- 恢复默认布局入口

### 6. 悬浮助手最小闭环

目标：悬浮助手成为常驻入口和继续入口，而不是装饰物。

本阶段必须具备：

- 关闭主窗口后退到托盘
- 可恢复主窗口
- 至少一个高频动作入口（如快速记笔记或快速加任务）
- 最近提醒或同步状态提示
- 明确的真正退出入口

### 7. 监督下自优化工作流

目标：让思忆集可以在明确授权下帮助维护思忆集自身或用户项目，但不能无边界自改。

本阶段可进入主线的工作流：

1. 用户提出改造需求。
2. 思忆集先生成变更提案。
3. 自动备份或生成回退点。
4. 在受控范围内执行修改。
5. 跑验证。
6. 输出变更报告。

这条链路的核心不是“AI 自动改代码”，而是“AI 走受控改造流水线”。

## 二、可以延期

这些能力有价值，但不应抢占当前主线资源。

### 1. 更复杂的数据源接入

- 更深层的 QQ / 微信接入
- 更多网站源自动接入
- 更复杂的系统行为感知
- 高权限网页下载、流媒体解析和大规模站点脚本维护

### 2. 更复杂的云能力

- 多设备强一致协同
- 复杂版本历史
- 分享、权限和多人协作

说明：当前更应该先把“设备概念、配对、冲突摘要、断网恢复、后台状态提示”做成可靠同步体验，而不是先追求更重的云协作能力。

### 3. 更复杂的布局能力

- 完全自由拖放布局
- 高复杂度容器嵌套
- 复杂主题系统

### 4. 更复杂的悬浮助手表现层

- 角色化外观
- 皮肤系统
- 更丰富的动画反馈

### 5. 更复杂的插件体系

- 开放第三方市场
- 插件签名分发平台
- 复杂插件依赖管理

## 三、当前阶段明确不做

这些方向不是永久不做，而是当前阶段不应进入主线，更不能抢占核心资源。

### 1. 无边界实时监控

不做“默认持续监控所有应用、所有窗口、所有聊天和所有行为”的产品路线。

原因：

- 风险过高
- 噪音过大
- 用户信任成本过高
- 技术实现脆弱

### 2. 无审查自治自修改

不做“思忆集自己决定如何修改自己，并直接落盘”的产品路线。

原因：

- 无法控制风险
- 很难建立回退边界
- 容易把项目带入不可维护状态

### 3. 开放插件市场

当前阶段插件只服务内部解耦和受控扩展，不开放市场生态。

### 4. 全量聊天平台化

不把思忆集做成 openclaw 或 nanobot 那样的多渠道消息平台。

当前阶段 QQ / 微信只作为“个人动态输入源候选”，不是产品主舞台。

### 5. 大而全个人云平台

不把“个人云”理解成直接替代百度网盘或做完整平台化套件。

当前阶段个人云只意味着：

- 有私人空间
- 有结构化整理
- 有上下文建议
- 有后端增强路径

## 四、阶段内优先级顺序

如果下一阶段只能排一条最小闭环，建议顺序如下：

1. 个人动态收件箱
2. 私人空间整理器
3. 当前内容整理能力
4. 上下文建议助手
5. 工作台化首页
6. 悬浮助手最小闭环
7. 监督下自优化工作流

原则：先让思忆集“持续收到输入并整理出来”，再让它“帮助推进下一步”，最后再让它“帮助改造自己”。

补充解释：

- 浏览器扩展、当前内容整理和私人空间落盘都服务于前 3 项，不应脱离这条主线单独扩张。
- 工作台化首页和悬浮助手的价值在于承接“继续入口”和“状态总览”，不是继续堆更多按钮。
- 监督下自优化必须建立在动作治理、审计、回滚和验证已成形的前提上。

## 五、完成定义

当以下条件同时成立时，可以认为下一阶段边界落地成功：

1. 用户可以稳定把日常输入收进思忆集。
2. 私人空间中已经形成 inbox、daily、insights 三层结构化沉淀。
3. 当前网页或文档可以通过统一动作得到摘要、要点和待行动项。
4. 思忆集的建议以低打扰方式出现，且可以转成任务或笔记。
5. 主界面已经是工作台容器，而不是继续堆积固定模块。
6. 关闭主窗口后，用户仍可通过托盘或悬浮入口继续使用思忆集。
7. 对思忆集自身或用户项目的自动修改，已经走到“提案、备份、执行、验证、报告”的受控链路。

## 六、参考项目反模式与约束补充（2026-04-06）

基于对 openclaw、nanobot、syncthing、七奈桌宠和 My-Neuro 的对标分析，下面补充当前阶段需要额外注意的反模式和约束。

### 从参考项目提炼的产品陷阱

| 陷阱 | 来源观察 | 思忆集约束 |
|------|---------|---------|
| 多渠道平台化 | openclaw 支持 23+ 消息渠道 | 思忆集不做消息平台，QQ/微信只作受控输入源候选 |
| 角色形态先行 | My-Neuro 的 Live2D 视觉效果吸引人但依赖底层稳定 | 先统一事件入口/字幕层/工具注册表，再升级表现层 |
| 无限自治 | nanobot 的轻量 loop 设计偏向个人场景 | 思忆集坚持"建议与执行分离"，任何自动动作都可审计、可回退 |
| 功能堆叠 | My-Neuro 同时做直播、游戏、RAG、TTS 等大量功能 | 思忆集先收口主线，不铺零散功能 |
| schema 不同步 | 历史教训：新增 tool_runs 链路后忘记更新 schema.sql | 数据库 schema 变更必须同步到测试临时库 |

### 当前阶段的质量债务边界

以下质量问题如果不先处理，"必须做"列表中的新功能会持续放大维护成本：

1. **ui_main_window.py 过重**（4800+ 行）：应先拆出工作台卡片、事件路由和布局管理，再开始新功能开发。
2. **assistant_autonomy.py 过重**：应拆成应用启动解析、规划、执行、Windows 环境探测等子模块。
3. **ui_translate_dialog.py 职责混合**：同时承载翻译、历史、单词本和同步，继续扩展前应先分离。
4. **测试覆盖薄弱**：当前主要依赖 smoke，仓储层、OAuth 刷新、同步冲突路径和浏览器桥接缺少单元/集成测试。
5. **SubagentManager 内存丢失**：`self._runs` 只存内存，进程崩溃时丢失，应增加持久化检查点。

### 补充原则

- 当前阶段"必须做"列表中的每一项在动手之前，都应先评估是否会加重上述质量债务。如果会，应先缩小改动范围或先做最小重构。
- 测试投入优先守住"可启动、可配置、可恢复、可诊断"，不先追求低频外观变体的覆盖。
